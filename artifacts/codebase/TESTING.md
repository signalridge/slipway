# Testing

- Question: What proof is needed for Workstream B's compact task result import
  and engine-owned run-version boundary?
- Existing command tests are old-protocol centered. `TestEvidenceTaskRecordsRuntimeEvidenceAndBuildsExecutionSummary`
  proves payload writing and execution summary materialization but calls
  `--run-summary-version`, `--task-kind`, and `--target-file` manually. Evidence:
  `cmd/evidence_task_test.go:18-95`.
- Existing mismatch and path tests cover useful failure behavior but must be
  adapted for result import and engine-derived run versions. Evidence:
  `cmd/evidence_task_test.go:388-470`.
- Run-summary ambiguity already has defensive behavior in wave-orchestration
  skill recording: task evidence files spanning multiple versions fail before
  wave-orchestration evidence is recorded. Evidence: `cmd/evidence.go:970-1024`.
- Resume/recovery tests show current task evidence can derive a resume wave
  index before execution-summary exists and can safely treat ambiguous run
  versions as not derivable. Evidence:
  `internal/engine/progression/wave_boundary_evidence_test.go:60-177`.
- S2 scope safety tests prove changed-file evidence is load-bearing. Missing
  `changed_files` for code tasks blocks visibly without wiping wave evidence.
  Evidence: `internal/engine/progression/wave_boundary_evidence_test.go:179-288`.
- Required new tests:
  - result-file happy path derives `task_kind`, `run_summary_version`,
    `target_files`, `captured_at`, and `freshness_inputs`;
  - result-file rejects executor-owned JSON containing `run_summary_version`,
    `task_kind`, `target_files`, `captured_at`, `freshness_inputs`, or
    `input_hash`;
  - invalid task IDs, verdicts, result paths, and changed-file paths fail
    closed without writing evidence;
  - active execution run version is generated by the engine and reused
    consistently across task evidence and wave-orchestration;
  - fresh re-execution/re-entry increments run version instead of reusing stale
    execution evidence;
  - template/toolgen/docs checks no longer teach agents the long task ledger
    flag protocol as the default path.
- Minimum verification commands from issue #297 B are package-focused Go tests,
  surface manifest check, and full `go test ./...` before S3 completion.
