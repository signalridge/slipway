##### STRUCTURAL CONTRADICTION — RESOLVED via R2 (intent text vs on-disk lattice)

The intent.md rescope text says every reviewer emits `context_origin:stage=review=<handle>`
(one shared "review" stage). Read literally against the on-disk lattice — keyed by
DISTINCT stage in `map[string]ContextParticipant` (context_attestation.go:194-216), with
templates emitting distinct `stage=spec` (spec template :147) / `stage=code` (code
template :117) — a single shared `review` stage would either collapse all reviewers to
one map entry (distinctness vacuous) or, for distinct handles under one stage key, hit the
per-record same-stage-conflict guard (context_attestation.go:72-77) and fail closed.

**Resolution — R2 (SELECTED by the user; adversary verdict = HOLDS): preserve the literal
`context_origin:stage=review=<handle>` wire token for ALL reviewers, but key the lattice
PARTICIPANT by recording-skill name, NOT by the `review` stage string.** The engine
assembles ONE single-handle `ContextParticipant` per review RECORD, keyed by its recording
skill; the skill-name keys (`spec-compliance-review`/`code-quality-review`/
`independent-review`/`security-review`, constants.go:10-13) are a DISJOINT namespace from
the stage-string keys `{executor, audit_origin, goal, closeout}`, so the flat participant
map has no key collision. `internal/model` adds ONLY the `StageContextReview` wire-token
constant; `CrossStageContextCollisions`/`participantsCollide`/`ContextParticipant`/
`ContextOriginHandlesFromVerification` are unchanged. Intra-review-set distinctness then
falls out of the EXISTING single-vs-single pair enumeration (`participantsCollide`
handle==handle, context_attestation.go:220-221) with zero new set logic, zero set∩set arm,
and no contributor-count side channel — the redesign lives entirely in the `authority.go`
feeder, subject to the TWO MANDATORY fail-closed conditions in Risks 1/2 and Q2 below.

R1 (review as an internally-distinct handle-SET) and R3 (set-fold review stage with a
`ReviewHandleSet{Handles,Contributors}` type) were REJECTED: both make `review` a
SET-valued lattice node, which (a) mutates the verified t-01..t-08 distinctness primitive
(a new set∩set arm in `participantsCollide`, breaking the "only executor is set-valued"
invariant at context_attestation.go:179-180), and (b) rest on an out-of-band contributor
count to detect duplicate handles before a Go-map set silently dedups them — exactly the
kind of invariant a future maintainer "simplifies" back into a fail-OPEN dedup, which the
project's fail-closed / self-optimization principles argue against shipping. R3 also
recommends a net-new `review_context_not_distinct` reason code (extra four-file contract +
docs churn). R2 needs neither.

### Patterns

The carried-over fail-closed parser idiom, advisory-on-light gate idiom,
absence-owned-by-required-skill-missing pattern, dual-surface-into-unresolved pattern, and
the generalize-target seam remain as previously documented. The review-SET adds two
established in-tree patterns:

1. **SINGLE-SOURCE-OF-TRUTH selection predicate** (the central review-SET pattern). The existing `code-quality-review` mechanism is the exact precedent: `WorkflowProfile.RequiresCodeQualityReview()` (workflow_profile.go:32-39) is a pure Change-derived predicate, and `FilterRequiredSkillsForWorkflowProfile` (skill.go:140-155) STRIPS `code-quality-review` from the required list when the predicate is false, so `evidence.go` (evidence.go:83-88) never fires `required_skill_missing` for an electively-absent reviewer. Security MUST follow the same shape: ONE selection decision consulted by `ResolveNextSkill` (routing fan-out) AND the required-skills filter (lattice participant + pass requirement) AND — under R2 — the lattice owned-set re-key and the participant loop. If the surfaces compute selection independently they can disagree (routed-but-not-required, required-but-not-routed → spurious `required_skill_missing`, or checked-but-unrequired in the lattice). NOTE the precedent keys on `WorkflowProfile` (coarse code/docs/research), NOT the richer Q1 security signals, so the *mechanism* (a post-`RequiredSkillsForState` conditional filter) is precedented but a `securitySelected` selector needs a NEW signal input plumbed into the call site (evidence.go:46-68 currently passes only needsDiscovery/workflowProfile/state/closeoutRequired/planSubSteps).

2. **Conditional-required registry membership.** `RequiredSkillsForStateWithRegistry` appends EVERY def whose `State` matches, gated only by `DiscoveryOnly`/`CloseoutConditional`/`PlanSubStep` (skill.go:100-138). An unconditional S3 registry entry is ALWAYS required → `required_skill_missing:security-review` whenever security is not dispatched (evidence.go:83-87; mechanically: one blocker appended per evaluation, polluting the blocker set ALONGSIDE the legitimately-required reviewers — fail-loud-but-wrong). So security needs a conditional mechanism analogous to `CloseoutConditional` (a `securitySelected` predicate, skill.go:17,105,125). `independent-review` registers unconditionally (always mandatory). The lattice already skips absent/non-passing records (`crossStageContextParticipants`, authority.go:502-571), so absent-optional security is doubly silent: not in `required`, and not a lattice participant.

3. **Distinctness asymmetry is preserved.** A present-passing review record yielding NO well-formed handle fails closed with `context_origin_handle_invalid` (authority.go:556-570) and short-circuits collision eval (~:627); a present-passing record whose handle collides fails closed with `cross_stage_context_not_distinct` (authority.go:609-641); only RECORD absence is delegated to `required_skill_missing`. This asymmetry already implements "security-selected-but-missing → required_skill_missing; security-selected-and-present-but-colliding → cross_stage_context_not_distinct".

### Risks

Carried-over risks (executor set-disjointness deadlock avoidance, plan-gate-cannot-see-executor, double-fire with P4, absence masking, retire-review_origin blast radius, fail-closed/self-stamp, lattice granularity, purity) all transfer. The review-SET (under R2) adds:

1. **REVIEW-OWNED-SET RE-KEY FAIL-OPEN (HIGH) — R2's first mandatory feeder condition.** Under R2 the literal `stage=review=` token is preserved (no stage-key collapse), but edge-keeping looks up BOTH endpoints in `ownedStages` (context_attestation.go:205-209). The current owned-set `crossStageContextOwnedReviewStages` (authority.go:464-469) is keyed by the STAGE STRINGS `{executor, audit_origin, spec, code}`. If the owned-set is NOT re-keyed to the dispatched review-SKILL-NAME keys, intra-review edges have no owned endpoint, are silently SKIPPED, and two reviewers can share a handle and PASS = FAIL-OPEN. Mitigation (MANDATORY): the owned-set AND the participant loop (authority.go:542-570) MUST be built from the SAME selected-review-skill slice so they cannot drift. Regression test: two reviewers with the identical `stage=review` handle → `cross_stage_context_not_distinct` on an OWNED edge.
2. **PARTICIPANT-SOURCE FAIL-OPEN (HIGH) — R2's second mandatory feeder condition.** Review participants MUST be sourced from `passingSkills` (the `EvaluateRequiredSkillsForChange`/`evidence.go` output, which contains only required+passing skills: evidence.go:61-68,83,133), NOT via the raw-disk `loadPresentPassingVerification` path that `audit_origin` uses (authority.go:528,595-607). The raw-disk path bypasses the required-filter and would resurrect an unselected-but-on-disk security record as a participant = FAIL-OPEN (checked-but-unrequired). Sourcing from `passingSkills` makes the lattice auto-inherit selection: unselected security → absent from `passingSkills` → no participant (silent); selected-missing → owned by `required_skill_missing`. The `securitySelected` predicate driving the registry/required-filter MUST be the identical value driving the participant loop + owned-set (one shared selected-skill list). Regression test: an unselected security record on disk → neither a participant nor a `required_skill_missing`.
3. **DUAL-SOURCE DIVERGENCE (HIGH).** If `ResolveNextSkill` and the required-skills filter derive security-dispatch from different logic, routing and gating disagree. The single Change/derivation-driven selection decision MUST be the sole input to both, exactly as `RequiresCodeQualityReview()` is for code-quality-review.
4. **PURITY TRAP for blast_radius (MEDIUM-HIGH).** `blast_radius` is NOT on `model.Change` — it is engine-derived (`control.DeriveControls`/`deriveBlastRadius`, derive.go:36-52,203-259) and lives on the governance `SignalSummary` (signal.go:60-63; readiness.go:55,194). `ResolveNextSkill` and `RequiredSkillsForStateWithRegistry` receive only the pure `Change` (skill_resolution.go:23; skill.go:100-132). So a selection contract that uses `blast_radius` CANNOT be a pure `internal/model` helper — it must be computed engine-side and threaded in (Q1 Option B), or restricted to `guardrail_domain`+`preset` (which DO ride `Change`: change.go:37). CAUTION: `model.Change` DOES carry two differently-named CALLER blast-radius fields — `CallerIndependentReviewBlastRadius`/`CallerWorktreeBlastRadius` (change.go:58-59) — coarse caller assertions, NOT the file-count-derived governance `BlastRadius`; do not conflate them. Putting governance-blast-radius logic in `internal/model` would force importing `engine/control` — an IMPORT CYCLE (`engine/control` already imports `model`: derive.go:9, config.go:3, evaluate.go:4), i.e. a compile-time error. (The `dependency_direction_test.go` architecture test only forbids `model`/`state` from importing the surface renderers cmd/tmpl/toolgen, so it would NOT itself catch this — the cycle compile error is the real guard.)
5. **SIGNATURE-CHANGE BLAST RADIUS (MEDIUM).** Threading a `securitySelected` param through `RequiredSkillsForStateWithRegistry` / `EvaluateRequiredSkillsForChange` touches several callers; enumerate and update all or risk compile failure under strict `changed_files ⊆ target_files`.
6. **STAGE-TOKEN PINNING (MEDIUM).** A lattice wire token not pinned in `TestContextAttestationPrefixConstsArePinned` drifts undetected. `StageContextReview` MUST be pinned. (R2 introduces no per-reviewer stage token, so there is no stage-vocabulary fan-out to pin beyond this one constant.)
7. **Q3 DISTINCTNESS is a CONTENT risk not caught by token tests (MEDIUM).** A token test proves only that each host emits its `context_origin` token; it does NOT prove the three reads differ. Making the three mandatory reviewers genuinely orthogonal (spec-trace vs engineering-quality vs unanchored-correctness/safety) requires removing the embedded base-reader bindings AND a prose/structure assertion that spec/code no longer contain the independent base-reader procedure. Quality-review's existing Safety section and independent's safety read can overlap — scope quality-Safety to pattern/convention safety and independent-Safety to diff-introduced regression safety.
8. **TOOLGEN GOLDEN CHURN (MEDIUM).** Promoting the two from static catalog SKILL.md to WorkflowOwned templated hosts changes `skill_tree_inventory.codex.golden` and the render-mode; `TestRenderCatalogSkillUsesTypedTemplatesForProductionSkill` (toolgen_test.go:785-804) depends on the STATIC independent-review SKILL.md and must be reconciled, not silently overwritten.
9. **#240-SELF MISCLASSIFICATION (LOW, disclosed residual).** #240 is `guardrail_domain=None` (intent.md:50) yet governance-sensitive. The domain arm deliberately does NOT auto-select security for it because #240's sensitivity is fail-closed-discipline, not code-level SAST risk; selection still follows the same blast-radius / strict-preset threshold as any other change. A future genuinely security-sensitive change mis-classified None with low blast radius would be missed by guardrail-only / blast-radius-only contracts; Option C's path heuristic (deferred) would close it.

### Options

#### Axis 1 (Executor join semantics) — UNCHANGED, carried over

- **Option A — set-disjointness (SELECTED).** Executor = the SET of all recorded non-empty task-handles (`ExecutorParticipantHandleSetFromVerification`, context_attestation.go:164-184); each stage handle must be ABSENT from that set; executor-internal distinctness stays owned by #239 P4 (`ExecutorAgentBlockers`, wave_sync.go:1015-1057). Empty set is vacuously silent; legitimately-shared executor handles dedup to one element; zero double-fire with P4. Blank `""` collapse values dropped.
- Option B — single-canonical (REJECTED): forces honest multi-subagent waves to fake one handle, manufactures deadlock for the common parallel case.
- Option C — all-distinct (REJECTED): duplicates P4's ownership of executor-internal structure and forbids the legitimately-shared case.

#### Axis 2 (Seam ownership) — UNCHANGED, carried over

- **Option A — earliest-resolvable-seam (SELECTED).** Each edge owned by the earliest seam where both endpoints are readable, keyed on `max(stage(a),stage(b))`: Plan gate owns the local `audit_origin != plan_origin` edge; Review authority owns the edges among `{executor, audit_origin, + the dispatched review participants}`; Ship authority owns the edges adding goal/closeout. Each edge once, earliest fail-closed timing, no double-fire. Note for the review-SET: the review-owned participant set is now CHANGE-DEPENDENT (adds independent always, security when dispatched), so both the review gate and the ship gate's base-participant load must compute the dispatched set from the SAME source `ResolveNextSkill`/required-skills uses, or the ship lattice reloads a stale fixed participant set.
- Option B — full-lattice-at-ship-only (REJECTED): defers S3-knowable violations to S4, weakens fail-closed timing.
- Option C — each-seam-recomputes (REJECTED): double-fires shared edges, violates exactly-once.

#### Axis 3 (S3 parallel review shape) — SUPERSEDED by the review-SET rescope

The prior **Axis 3 = "fixed parallel pair, no new host" is SUPERSEDED by the 2026-06-17 review-SET rescope.** The carried-over rationale (the engine never enforced spec→code ordering: authority.go:93-102 evaluates both reviews together; `closeoutChainOrderBlockers` is cross-stage only) STILL HOLDS and is the foundation the variable set builds on, but "no new host / fixed pair" no longer applies: independent-review becomes a mandatory standalone host and security-review an optional standalone host, so S3 is a VARIABLE SET. Q1/Q2/Q3 below replace Axis 3, and the shared-review-stage redesign (R1/R2/R3) resolves how a single `stage=review` token expresses N-reviewer distinctness.

#### Shared-review-stage redesign (R1 / R2 / R3) — R2 SELECTED

How does a literal shared `context_origin:stage=review=<handle>` express mutual distinctness across 3-4 reviewers, given the on-disk lattice keys participants by distinct stage?

- **R1 — review as an internally-distinct handle-SET (REJECTED).** One `review` lattice node valued as the SET of reviewer handles; adds a `RequireSetDistinct` bool to `ContextParticipant` and a set-vs-set arm to `participantsCollide`; intra-set distinctness detected by counting occurrences before set-collapse. Fail-closed ONLY if the feeder counts handles before dedup — the headline property rests on NOT trusting a Go-map set (naive dedup = fail-OPEN), an out-of-band count easy to regress. Mutates the verified distinctness primitive and breaks the "only executor is set-valued" invariant.
- **R2 — literal stage=review token, participant keyed by recording skill (SELECTED; adversary verdict = HOLDS).** All reviewers emit literal `context_origin:stage=review=<handle>`; the engine assembles ONE single-handle participant per review RECORD keyed by its recording skill. Intra-review-set AND review-vs-other distinctness fall out of the existing pairwise edge enumeration with ZERO new set logic; the `review` stage string stays a pure wire/parse token, never a participant key. `internal/model` change = ONLY `+StageContextReview`; `CrossStageContextCollisions`/`participantsCollide`/`ContextParticipant`/`ContextOriginHandlesFromVerification` UNCHANGED and stdlib-pure. No new reason code (reuse `context_origin_handle_invalid` + `cross_stage_context_not_distinct`; the detail now names skills, e.g. `code-quality-review|independent-review`, strictly clearer than R1's `review|review` self-pair).
