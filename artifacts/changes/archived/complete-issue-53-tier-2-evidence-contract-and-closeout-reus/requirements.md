# Requirements
## Project Context
- Tech Stack: Go CLI, Cobra, Slipway governance runtime
- Conventions: command metadata in `internal/toolgen`, command wiring in `cmd/root.go`, compact YAML verification records, flat runtime task evidence JSON
- Test Command: go test ./...
- Build Command: go build ./...
- Languages: Go

## Requirements

### Requirement: Supported task evidence writer
REQ-001: The system MUST expose a supported CLI command for recording governed task evidence so hosts no longer hand-write `.git/slipway/runtime/changes/<slug>/evidence/tasks/*.json`.

#### Scenario: Task evidence command writes consumer-compatible evidence
GIVEN an active governed change has wave-orchestration run version `1`
WHEN the host invokes the supported task evidence command for `task_id=t-01`
THEN Slipway writes a flat JSON task evidence file under the canonical runtime evidence directory.
AND `progression.ParseTaskEvidence` can parse that file for run summary version `1`.

### Requirement: Controlled task evidence fields
REQ-002: The task evidence command MUST compute or validate controlled runtime fields, including `run_summary_version`, `captured_at`, `freshness_inputs`, `task_kind`, `verdict`, `evidence_ref`, and task blockers.

#### Scenario: Freshness inputs are generated by Slipway
GIVEN a task evidence command request omits freshness inputs
WHEN the command records the evidence
THEN the stored payload contains `state.ExpectedExecutionTaskFreshnessInputs(change, run_summary_version, task_id)`.
AND caller-supplied stale or cross-change freshness inputs are not required or trusted.

#### Scenario: Invalid task verdicts fail closed
GIVEN a task evidence command request names an invalid verdict
WHEN the command validates the payload
THEN the command exits with an actionable validation error and does not write task evidence.

### Requirement: Wave orchestration contract records runtime ledger
REQ-003: The wave-orchestration host contract MUST instruct executors/orchestrators to use the supported task evidence command before writing passing wave-orchestration evidence, so `slipway run` can build execution summaries without manual JSON edits.

#### Scenario: Generated skill guidance includes the supported surface
GIVEN Slipway renders the wave-orchestration skill
WHEN a host reads the dispatch/write-evidence contract
THEN the contract names `slipway evidence task` and the required task evidence fields.

### Requirement: Repair distinguishes source task evidence from derived summaries
REQ-004: `slipway repair` MUST continue rebuilding stale or unreadable execution summaries from valid source task evidence and MUST report missing source task evidence as non-repairable instead of fabricating summaries.

#### Scenario: Repair rebuilds from supported source evidence
GIVEN a governed change has passing wave-orchestration evidence and valid task evidence written through the supported surface
WHEN `slipway repair --json` runs after `execution-summary.yaml` is missing or stale without planning drift
THEN repair rebuilds `execution-summary.yaml` from the task evidence.

#### Scenario: Repair reports missing source evidence
GIVEN a governed change has passing wave-orchestration evidence but no task evidence for that run version
WHEN `slipway repair --json` runs
THEN repair reports a non-repairable finding naming missing task evidence and the supported task evidence command.

### Requirement: Final-closeout reuse is explicit and fail-closed
REQ-005: Final-closeout MUST support an explicit goal-verification reuse branch only when goal-verification proof is fresh for the current run/content state, and MUST reject stale or mismatched reuse.

#### Scenario: Matching reuse passes
GIVEN goal-verification passes for the latest execution run and execution-summary freshness is fresh
WHEN final-closeout records explicit reuse references for that goal-verification run
THEN ship readiness accepts the final-closeout record without requiring duplicated full-suite proof.

#### Scenario: Mismatched reuse blocks ship
GIVEN final-closeout records goal-verification reuse for a different run version, missing goal-verification record, or stale execution-summary content state
WHEN ship readiness evaluates required skills
THEN the ship gate reports an actionable closeout reuse blocker.

### Requirement: Regression coverage and docs/templates
REQ-006: The change MUST add focused regressions and update generated host/user-facing guidance for the new evidence command and final-closeout reuse contract.

#### Scenario: Focused and full verification pass
GIVEN the implementation is complete
WHEN targeted tests, `go test ./...`, `go build ./...`, `go vet ./...`, `git diff --check`, and `slipway validate --json` run
THEN all checks pass before the workflow stops short of `slipway done`.
