# Decision

## Alternatives Considered
- **Approach A — engine-enforced fail-closed evidence gates (C0–C3) + non-blocking widening advisories (C5) + doc alignment (C4).** The engine turns the host's already-recorded evidence (`changed_files`/`target_files`, `dispatch_mode`, `executor_agent`) into hard gates, without growing an in-engine spawn runtime. Tradeoffs: smallest clean design that reuses the planner's existing conflict predicates; C2 is a deliberate breaking change; widening stays advisory (human judgment in plan-audit). Selected.
- **Approach B — per-executor worktree isolation + worktree proof tokens (gsd-core model).** Would make same-wave collisions structurally impossible. Rejected: the user has deliberately rejected per-task isolation (isolation stays change-level); it adds a large new runtime surface and contradicts the established "engine governs, host executes" architecture.
- **Approach C — gsd-core-style fail-soft auto-downgrade (overlapping wave silently drops to sequential and continues).** Never blocks. Rejected: normalizes silent loss of safety, violates the project's fail-closed principle for sensitive work, and reintroduces exactly the silent-inference disease C2 removes.

## Selected Approach
Approach A. It satisfies the Approved Summary (engine-enforced scope-escape + overlap audit, dispatch-evidence and executor-agent fail-closed gates, and non-blocking narrowing advisories) with the smallest clean, host-driven design. All new blockers fold into the single existing assembly point in `evaluateGovernedWaveExecution` (`internal/engine/progression/wave_sync.go` ~:173-180) under the existing `len(planDriftBlockers)==0` suppression guard, flow out through `ExecutionSummary.OpenBlockers`, and surface via `validate`/`next`/`status --json` — no new persistence or JSON pipeline. C1 reuses the planner's conflict predicates via a single exported `wave.TargetCoversPath`, so "coverage" and "conflict" share one implementation. C0 is the root fix: because `WaveDispatchMode.IsValid()` switches on the constant, changing `WaveDispatchParallel`'s value to `parallel_subagents` makes the engine read the exact token the generated skill already instructs the host to write. The engine signals, records, and gates but never executes work itself.

## Interfaces and Data Flow
- **Changed public contract (sensitive — `external_api_contracts`)**: `WaveRun.dispatch_mode` output value `parallel` → `parallel_subagents`; the `validate`/`next`/`status --json` blocker-code set gains `task_changed_file_scope_escape`, `parallel_wave_changed_file_overlap`, `dispatch_mode_absent_on_started_parallel_wave`, `executor_agent_missing`; the wave-plan view gains a view-only `advisories` array (omitempty).
- **New internal interfaces**: `wave.TargetCoversPath(targets, file) bool` (exported wrapper over existing predicates); `wave.AnalyzeWaveNarrowingCauses(nodes) []string`; `model.ExecutorAgentHandlesFromVerification(record)`; four blocker functions in `wave_sync.go`.
- **Data flow (unchanged otherwise)**: `tasks.md` → `PlanWaves` → `MaterializeWavePlan` (`Parallel`) → `wave-plan.yaml` → `cmd/next*` view (+ advisories) → generated skill → host fan-out → verification references → `BuildWaveRuns` (`dispatch_mode`, no inference) → `evaluateGovernedWaveExecution` (new gates) → `OpenBlockers`.
- The new `advisories` field is computed in the view layer only and is excluded from the `tasks_plan_hash` family, so freshness is unaffected.

## Rollout and Rollback
- Rollout: single cohesive change merged through this governed flow; generated host surfaces regenerated via toolgen so code, skill, and docs ship aligned. Dogfood with a worktree-built binary (installed 0.22.1 lags and lacks these gates).
- Rollback: revert the change's commit(s); the new blocker codes and the `dispatch_mode` value revert with the diff. No data migration is involved (runtime dispatch state is regenerated each sync). Rollback verification: `go build ./... && go test ./...` on the reverted tree, and `slipway next --json` shows neither the new blockers nor `wave_plan.advisories`.

## Risk
- **C2 is a deliberate breaking change (medium)**: removing the silent-parallel inference blocks in-flight started parallel waves that lack a `dispatch_mode` token until the host records it. Mitigated by C0 (the literal now matches the skill the host already follows) and C4 (docs); must be called out in the PR body. Reversible.
- **C1 audit is post-hoc (medium, inherent to change-level isolation)**: a parallel wave's overlap/scope-escape is only detected when its evidence is evaluated — by then the wave may already be committed into the shared worktree. Recovery = fix `target_files` and re-record evidence, or rescope. Strictly better than today's zero audit; this is the accepted cost of rejecting per-executor isolation.
- **C1 overlap audit intentionally keys off plan-parallel, not recorded dispatch (low)**: `parallel_wave_changed_file_overlap` is evaluated for waves whose effective plan says `parallel: true`, even if the host later records `degraded_sequential`. This is a fail-closed plan-quality choice: a wave that the plan claims is safe to parallelize but whose completed tasks share a changed file is not a safe parallel plan and should be rescoped, rather than silently passing because one execution happened sequentially.
- **C3 cannot prove real parallelism (low, inherent to "engine doesn't spawn")**: a coordinator can write N distinct fake handles then run inline-serial. C3 only raises the bar (requires one handle per planned task); it is honestly scoped as evidence-completeness, not a parallelism proof.
- **C1 false-positive risk (low)**: path aliasing — mitigated by normalizing both sides through `model.NormalizePublicPath` before comparison.
- **Contract-test breakage (low)**: regenerated skill/docs must keep `wave_isolation_content_test.go` green (no `engine rejects` / `engine-level rejection`); README/adapter contract tests must stay green.
