# Mutation Configuration

Defaults are a starting point, not a finish line. The wrong mutator set produces thousands of equivalent
mutants that drown the signal; the wrong exclusions let generated code dominate the score. This note
covers the two decisions worth making deliberately: which mutators to run, and which code to exclude.

## Default Mutator Sets by Language

### Python — mutmut, cosmic-ray

- **mutmut** ships a small, opinionated mutator set: number literals, boolean literals, comparison
  operator flips, `and`/`or` flips, arithmetic operator flips, `break`/`continue` removals. Low noise,
  good default.
- **cosmic-ray** is pluggable. A reasonable starting set: arithmetic, comparison, boolean, return-value,
  exception-swallow. Expand only when the default score converges.

### JavaScript / TypeScript — Stryker

Stryker's default mutator set covers arithmetic, equality, logical, update (`++`/`--`), conditional
boundary, string-literal, array-declaration, block-statement. Disable `StringLiteral` early — it
generates large numbers of mutants that are either trivially killed (user-facing strings with snapshot
tests) or never killed (log messages).

### JVM — PIT

PIT ships `DEFAULTS` and `STRONGER` groups. Start with `DEFAULTS`. The `STRONGER` group adds
`EXPERIMENTAL_SWITCH`, `REMOVE_CONDITIONALS`, `REMOVE_INCREMENTS` — each produces high-value mutants but
also more equivalents. Enable per-package once the `DEFAULTS` score is above ~70%.

### C++ — mull

mull operates on LLVM IR. Useful defaults: `cxx_arithmetic`, `cxx_comparison`, `cxx_boundary`,
`cxx_remove_void_call`, `cxx_remove_negation`. Skip `cxx_replace_numeric_constant` for numeric-heavy code
until you have targeted tests — it dominates mutant counts in numerical kernels.

### Go — go-mutesting

Default mutators: conditional branch, integer increment/decrement, boolean expression. Go's lack of
generics in older versions produced many typed duplicates; with generics this is less of a concern. The
ecosystem is less mature than the above — expect to hand-curate the mutator list per repo.

## Mutators to Disable for Noise

Some mutators are either semantically equivalent or kill each other's signal. Disable them unless you have
tests specifically written against the behavior they change.

- **Boolean-literal flips in logging.** `log.debug("enabled")` → `log.debug("disabled")` produces mutants
  that tests correctly ignore. Exclude logger namespaces.
- **Number-literal flips in metrics labels.** `counter("requests", 1)` → `counter("requests", 0)`.
  Exclude metrics namespaces or add a mutator filter for `inc`/`add`/`observe` calls where the literal is
  a label value, not a magnitude.
- **String-literal mutations for user-facing text.** Your tests rightly do not assert on every comma in an
  error message. Either exclude string-literal mutators globally or scope them to machine-consumed
  strings (config keys, protocol constants).
- **`return true` / `return false` on trivially-unreachable branches.** If coverage shows the branch is
  dead, fix the coverage gap; do not let the mutant frame it as a test weakness.
- **Assertion-message mutators.** `assert x > 0, "must be positive"` vs `"bad value"`. The assertion
  still fires; the message is not a behavioral contract.

## Equivalent Mutants

A mutant is equivalent if no observable behavior changes regardless of input. Equivalents are the tax of
mutation testing; aim to keep them below ~5% of the surviving mutant population.

Common sources:

- **Integer overflow semantics.** `i + 0` → `i`, `i * 1` → `i`, in languages with defined arithmetic.
- **Short-circuit reordering.** `a && b` → `b && a` when both sides are pure.
- **Dead code past a `return`.** Mutating statements after an unconditional return.
- **Symmetric predicates.** `a == b` → `b == a`.

Most frameworks offer an `@AssumeEquivalent` or suppression annotation; use it in-source rather than in
config, so the next reader sees the reason.

## Handling Auto-Generated Code

Generated code must be excluded. It is usually untested at the generator output level (tests live against
the generator itself), and its mutation score drags the average down while telling you nothing actionable.

- Path patterns: `**/generated/**`, `**/*.pb.go`, `**/*_generated.ts`, `build/generated-src/**`,
  `*.designer.cs`, `*.g.dart`.
- Annotation-based: exclude classes with `@Generated`, `// Code generated by`, `#region Designer
  generated code`.
- Build-output trees: `build/`, `dist/`, `target/classes/`, `out/` — exclude via the config file.

Review the exclusion list on every mutation-framework upgrade; generator outputs drift.

## Config-File Exclusion Patterns

Keep exclusions declarative, versioned, and reviewed. Do not bury them in CI scripts or environment
variables — they are specification, not deployment trivia.

- Mirror the test-coverage exclusions where possible; divergence means either mutation or coverage is
  lying.
- Exclude whole directories for migrations, fixtures, vendored dependencies, third-party bundled code.
- Exclude file patterns for generated code (above).
- Allow per-file annotation overrides for the odd case where a generated file has hand-edits — the
  annotation makes the exception auditable.

## Mutation-Score Targets by Layer

A single repo-wide mutation score is a blunt instrument. Different layers earn different targets.

| Layer                          | Target mutation score | Rationale                                      |
|--------------------------------|-----------------------|------------------------------------------------|
| Core domain / algorithms       | ≥ 80%                 | Pure, testable, the reason the product works.  |
| Adapters / IO boundaries       | ≥ 60%                 | Testable with fakes; some mutants equivalent.  |
| Glue / wiring / main functions | ignore                | Measuring here produces more noise than signal |
| Config / constants             | ignore                | Values, not behavior.                          |
| Generated code                 | excluded              | Not reviewed by humans.                        |
| Vendored / third-party         | excluded              | Not your test surface.                         |

A rubric beats a target. Report mutation score per layer and fail CI on regressions *within a layer*, not
on the aggregate. Aggregate scores hide dramatic drops in the core behind stable scores in the glue.

## Configuration Review Cadence

- **Per PR:** do not touch config. Signal must be comparable across runs.
- **Per quarter:** review the top 10 mutators by surviving-mutant count. Disable or re-scope noisy ones;
  expand coverage for high-signal ones.
- **Per major framework upgrade:** re-validate the exclusion list against new default mutators.
- **On generator upgrades:** refresh generated-code exclusions.

Stable config is the difference between a mutation score that trends meaningfully over months and a
number that fluctuates with every toolchain bump.
