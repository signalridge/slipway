# Authentication — secure defaults

## Password storage
Accept one of these hash families; reject anything else as legacy:

| algorithm | required parameters | notes |
|-----------|--------------------|-------|
| Argon2id | memory ≥ 19 MiB, iterations ≥ 2, parallelism 1 | preferred |
| scrypt   | N ≥ 2^17, r = 8, p = 1 | OK when Argon2 unavailable |
| bcrypt   | work factor ≥ 10 (12+ preferred); cap password at 72 bytes | legacy systems |
| PBKDF2   | HMAC-SHA-256, iterations ≥ 600 000 | FIPS-required contexts only |

Salts are unique per password, ≥ 128 bits, generated by the library's
CSPRNG. Never use MD5, SHA1, plain SHA-256, or rot-your-own key
stretching.

## Password policy
- Allow all printable characters, including spaces and Unicode.
- No mandatory complexity rules; no scheduled expiry.
- Minimum length 15 (8 with MFA). Maximum length ≥ 64.
- Check against breach corpora (HIBP or offline equivalent).
- Rate-limit and lock-out after failed attempts, on both username and
  source IP dimensions.

## Session management
- Session IDs generated with a CSPRNG; ≥ 128 bits of entropy.
- Never put session state in the URL. Cookies only:
  `Secure`, `HttpOnly`, `SameSite=Lax` (or `Strict` for privileged
  flows), `Path=/`, no unnecessary `Domain` scope.
- Absolute TTL ≤ 12 hours for standard users, ≤ 30 minutes for
  admins; idle TTL ≤ 30 minutes.
- Rotate the session ID on authentication, privilege escalation, and
  MFA upgrade. Invalidate server-side on logout.

## Multi-factor authentication
- Prefer phishing-resistant factors: WebAuthn / FIDO2 passkeys.
- TOTP is the acceptable second tier; SMS and email are last resort.
- MFA challenges re-evaluate on password change, on new device, and
  for privileged operations (re-auth, not just session presence).
- Recovery codes are one-time, displayed once, stored hashed.

## Credential transmission
- TLS 1.2+ with modern cipher suites; HSTS enabled with
  `max-age ≥ 31 536 000`.
- Basic auth only over TLS; prefer bearer tokens or cookie sessions.
- Reject credentials in query strings; they leak into logs and
  referrers.

## Account recovery
- Recovery flows must not leak whether an account exists. Generic
  "if this address is registered, we sent an email" responses.
- Reset tokens: 128-bit CSPRNG, single-use, TTL ≤ 1 hour, bound to
  user id and requesting device fingerprint where possible.
- Require re-authentication (not just the reset link) before the user
  can change the primary email or disable MFA.

## Review cues
- Any hand-rolled hashing, signing, or token generation is a blocker
  unless paired with documented rationale from a cryptographer-level
  reviewer.
- "Password policy says 90-day expiry" is a blocker; modern guidance
  rejects forced rotation.
- "We generate a token with `Math.random()` / `rand()`" is a blocker;
  CSPRNG only.
- Session-fixation vectors (no rotation on auth) are blockers.

## Anti-patterns
- Storing password hints.
- Returning full password strength feedback that discloses the
  server's allowed character set.
- Using email verification alone as MFA.
- Reusing session tokens across trust boundaries (public ⇄ admin).
