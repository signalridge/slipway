# ADR-0003: Scope `slipway-workflow` to Slipway functions, not skill catalogs

## Status

Accepted — 2026-07-18. This ADR refines ADR-0002. It supersedes ADR-0002 only where that record describes `slipway-workflow` as an idea-to-draft first-half orchestrator, assumes exactly two downstream authorization boundaries, or suggests artifact-producing external primitives as part of its normal method. ADR-0002's decision to add the seventh generated capability and its no-router, no-runtime-state constraints remain accepted.

## Context

ADR-0002 established that a seventh, explicit, host-side capability could bridge a rough idea to Propose without adding a CLI command, protocol field, Run state, journal, launcher, or global router. Its first-half framing solved that entry case but left the capability narrower than the product contract it was meant to make navigable.

Issue #434 already defines distinct user-owned functions for standalone clarification, publication, Objective decomposition, issue-backed or ad-hoc Run startup, Run recovery, direct implementation, and advisory review. A user may enter with an existing Objective, a valid Change, a paused Run, or findings from an ended Run. Sending every entry through idea clarification and drafting would duplicate completed work, while calling installed third-party skills would turn the capability into the general orchestrator that the design explicitly rejects.

The phrase "two authorization boundaries" is also too rigid. A Change path crosses Propose and Run; an Objective path additionally crosses Decompose; a valid existing Change may cross only Run; and each failed or ambiguous external-write operation must stay in its own reconciliation step. The durable invariant is not a boundary count. It is that the workflow owns no external write or Run-start authority and never advances across a user-owned capability boundary by itself.

## Decision drivers

- Make every relevant function defined by issue #434 reachable from an explicit workflow entry without forcing unnecessary earlier stages.
- Coordinate Slipway's product lifecycle, not the host's arbitrary installed skills.
- Preserve each existing capability's authority, confirmation, recovery, and ownership contract.
- Keep the seventh capability stateless, read-only, self-contained, additive, and host-side.
- Preserve the generated bytes of the six pre-existing capabilities so six-to-seven refresh remains ownership-safe.

## Decision

`slipway-workflow` is a stateless lifecycle navigator for Slipway's own functions:

1. It inspects the observed starting point and, when the user chooses to continue, names exactly one immediate next explicit capability. Rough ideas or non-authoritative planning artifacts route to Propose after draft synthesis; a decision-only request whose endpoint is a bounded summary routes to standalone Clarify; Objectives route to Decompose; structurally valid Changes route to Run; an explicitly private, tiny, urgent, offline, or untracked bounded goal routes to a new ad-hoc Run; active Runs route back to their exact current Action and control rules; paused or stopped Runs route to structured recovery; failed, partial, or ambiguous publication returns all available receipt/operation/item/revision facts to the originating Propose or Decompose owner, which decides same-receipt reconciliation or a contract-required fresh preview and confirmation; ended Runs or findings allow no further action, new tracked scope, a new issue-backed Run after fresh Change fetch and attestation, or a new ad-hoc Run on a sharpened untracked goal. Choosing no further action is a valid terminal outcome and names no capability.
2. It may perform repository investigation, a one-question-at-a-time decision interview, and Change-or-Objective draft synthesis when the current stage needs them. It skips those activities when the input is already structurally valid for a later stage.
3. It never invokes a sibling `slipway-*` capability. It does not discover, rank, or dispatch the host's skill catalog. Its only optional external accelerator is an already-installed, model-invocable `/grilling` primitive; absence never blocks the route or triggers installation. ADRs, reports, prototypes, and persistent maps remain separate, explicitly invoked detours whose outputs may later be planning input.
4. The workflow itself is read-only and creates no Run, journal, cross-stage cursor, publication plan, Issue, prototype, report, or code change. Every external-write operation retains its operation-scoped current confirmation, and every Run start or resume remains explicit. Failed, partial, or ambiguous publication remains in reconciliation.
5. Only a structurally valid, self-contained `change/v2` Issue whose manifest-addressed chapters pass source validation may route to issue-backed Run. A bare Issue number is a host locator, not CLI source. For a new Run, when no initial budget override is supplied, the host states and uses the contract default of `8`; an override remains `1..1000`. Resume instead preserves a positive remainder or replenishes zero to `max(initial_budget, 3)` when the resume mutation actually applies.
6. Standalone Clarify, Implement, and Review remain available when the user explicitly chooses their standalone outcome, but the workflow does not insert Clarify as a mandatory drafting stage or use direct Implement/Review as shortcuts around a managed Change-to-Run route. Once a Run starts, the Run owns Orient, conditional Clarify, Implement, advisory Review, Summary, current Action control, and structured recovery. For an active Action, submit and skip are structured variants; stop is a public command, while take-over and reorder first stop the Run and hand control back.
7. This refinement changes generated workflow instructions, user documentation, tests, and acceptance scenarios only. It adds no CLI command, protocol field, Action kind, Run state, journal entry, governance gate, network behavior, credential handling, or mutation of the six pre-existing capability templates.

## Consequences

Users get one explicit orientation surface for all lifecycle starting points covered by issue #434, while each actual operation remains independently user-owned. The workflow can recommend a complete conditional route but stops at the first capability boundary, so a failed publication cannot accidentally become execution. Advisory findings remain non-gating: the user may accept them and choose no further Slipway action.

The generated capability remains usable without Matt Pocock's skills. Installations that do include them gain only the optional model-reachable interview primitive; their user-only front doors stay human-only. This avoids an undeclared dependency closure and prevents the seventh capability from becoming an ambient launcher.

Tests and acceptance evidence must cover rough ideas, standalone decision-only clarification, Objectives, valid Changes, explicit ad-hoc starts, active Runs, paused or stopped Runs, ended issue-backed and ad-hoc Runs, standalone Implement/Review routing, absence and presence of `/grilling`, new-Run default and resume budget behavior, reconciliation stops, and the prohibition on general skill dispatch. They remain evidence of observable behavior, not completion or release certification.

The authoritative Chinese contract in live issue #434 was reconciled with the seventh capability and this lifecycle scope on 2026-07-18; the accepted ADRs now preserve the rationale instead of compensating for a live exact-six mismatch.

## Rejected alternatives

- **Orchestrate every installed skill.** Skill discovery and dispatch would create an unbounded router, make behavior installation-dependent, and exceed issue #434's product surface.
- **Always restart from a rough idea.** Re-drafting a valid Objective, Change, or Run recovery state would discard authoritative lifecycle facts and add needless user work.
- **Invoke Propose, Decompose, or Run automatically.** Those capabilities are explicitly user-invoked and own distinct write, reconciliation, source, or execution authority.
- **Add a persistent workflow cursor.** Existing Objective, Change, source, and Run records already own durable state. Another cursor would introduce competing recovery semantics.
- **Modify the original six templates to add handoff text.** Existing managed files may be user-modified and are intentionally byte-compatible across the six-to-seven upgrade; the new navigator can express the complete route without rewriting them.
