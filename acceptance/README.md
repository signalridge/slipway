# Issue #434 acceptance evidence matrix

This matrix maps all 35 normative scenarios to reproducible artifacts and honest evidence gaps. It is an evidence index, not a release verdict, Issue status, Run routing input, Review result, or CLI progression input.

The executable Shell and native Windows assets are the repository's black-box end-to-end tests against a built Slipway binary. Fixtures, validators, prompt scenarios, and transcript instructions support the broader acceptance evidence and are not all E2E tests.

Evidence labels have exact meanings:

- **C** — deterministic Go contract/property/race test or static generated-template assertion; proves only CLI/core observable semantics or template text.
- **S** — Shell acceptance under this directory that invokes a built Slipway binary.
- **G** — isolated live GitHub.com user-owned fixture. The local publication fault harness is H/G-adjacent, not G.
- **H** — a real sanitized Claude/Codex/Pi prompt transcript plus evaluator notes.
- **W** — native Windows execution through both `cmd.exe` and PowerShell.
- **R** — documentation, website, package, or release validation.

C cannot substitute for H. A fake endpoint or this deterministic harness cannot substitute for live G. A Windows cross-build or static script scan cannot substitute for native W. `available` means the linked executable/static artifact exists in this tree; `not collected` means required runtime evidence has not been recorded; `external` means collection needs a protected account, host, or native platform outside this local run. Missing evidence is uncertainty and never controls CLI progression.

## R validation package

R does not replace any row's required C/S/G/H/W evidence. Repository-side R artifacts include the three-locale [documentation](../docs/en/index.md), authoritative [Chinese contract](../docs/zh/reference/product-contract.md), [website navigation](../website/astro.config.mjs), stdlib [link checker](link_check.py), reproducible [release artifact checker](release_artifacts.py), and CI/docs/release workflow evidence collectors. The checker assets and workflow wiring are **available**. A successful invocation establishes only that invocation's R result; command outcomes belong in the implementation handoff and are not a delivery certification.

## Commands

From the repository root, with a temporary binary outside the tree:

```bash
acceptance_tmp="$(mktemp -d)"
trap 'rm -rf "$acceptance_tmp"' EXIT
go build -o "$acceptance_tmp/slipway-acceptance-bin" .
SLIPWAY_BIN="$acceptance_tmp/slipway-acceptance-bin" sh acceptance/machine-protocol.sh
SLIPWAY_BIN="$acceptance_tmp/slipway-acceptance-bin" sh acceptance/adapters.sh
python3 -I acceptance/github_publication_fault_harness.py
python3 -I acceptance/link_check.py --self-test
# After `npm --prefix website ci && npm --prefix website run build`:
python3 -I acceptance/link_check.py --require-site
# After a GoReleaser snapshot populated dist/:
python3 -I acceptance/release_artifacts.py --dist dist
```

See [Windows assets](windows/README.md), [live GitHub instructions](live-github/README.md), and the [sanitized transcript format](transcripts/README.md) for evidence that cannot be collected on an ordinary macOS/Linux run.
